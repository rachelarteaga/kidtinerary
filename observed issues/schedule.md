# Schedule

Shared/public schedule view.

## Issues

- ~~**Live share link returns 404 for recipient** — Rachel sent a live dashboard/schedule link to a friend; friend clicked and got a 404 page.~~ Fixed 2026-04-24 via PR #17 (`fix/share-link-404`): anonymous viewers of `/schedule/[token]` unblocked — root cause was RLS filtering on the share-lookup query; refactored into a `SECURITY DEFINER` RPC (`get_shared_planner_by_token`, migration 029) with token-scoped access, extracted the result-mapping logic into `src/lib/queries-share-mapper.ts`, and added test coverage.
- ~~**Shared planner shows fewer kids than the owner has** — friend shared her planner with two kids checked; when Rachel received the link, only one kid appeared.~~ Diagnosed + fixed 2026-05-09: the DB share record for that token literally only had one `kid_id` — the recipient view + RPC were correct. Root cause was a stale `useState` lazy initializer in [share-planner-modal.tsx](src/components/planner/share-planner-modal.tsx): the `selected` Set was seeded from the `kids` prop at first mount and never re-initialized, so kids added to the planner *after* the modal first mounted stayed silently unchecked even though the UI implied "share all kids". Fix: (1) parent now conditionally renders the modal (`{shareOpen && ...}`) so each open is a fresh mount; (2) `selected` / `includeCost` / `includeBlocks` seed from the live `shared_schedules` row when one exists, falling back to all-kids when not; (3) [addKidToPlanner](src/lib/actions.ts:1173) now mirrors the new kid into any active share for that planner as a defensive belt-and-suspenders for the case where a kid joins the planner *after* the share was created.
